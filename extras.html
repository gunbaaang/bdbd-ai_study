번외: AI별 결과 비교

같은 요청을 여러 AI에게 던졌을 때 방향이 어떻게 달라지는지 정리한 실험 기록입니다. 본 실습 흐름과는 별개로 참고용으로 봐주세요.

← 실습 가이드로 돌아가기

CASE 1 : 같은 프롬프트 입력 후 화면 결과 차이

같은 "스터디 모임 모집 서비스" 프롬프트를 입력했지만,
AI별로 화면을 구성하는 방향과 강조 기능에 차이 발생함

요약하면

Codex

스터디 개설부터 신청자 관리까지 포함해, 운영자가 서비스를 관리하는 화면 방향으로 구성

Claude

사용자가 스터디를 찾고 신청하는 흐름에 집중해, 일반 사용자의 탐색 화면 방향으로 구성

※ 다만 이는 하나의 실험 사례이며, AI별 결과 방향은 요청 맥락과 프롬프트 조건에 따라 달라질 수 있다.

CASE 2: 같은 요청으로 확인한 AI별 프롬프트 차이

같은 "스터디 모임 모집 서비스" 요청을 Claude와 Codex에 각각 입력해본 결과,
프롬프트 구성 방향에 차이 발생함

요약하면

Claude

기술 스택, DB, 인증, 배포, 개발 순서까지 포함한 실제 개발형 프롬프트로 구성

Codex

더미데이터, 화면 흐름, UI 구성 중심의 기획 검증형 프롬프트로 구성

항목별 세부 비교

구분 Codex Claude
전체 범위 판단 일단 화면으로만 확인해보는 시제품 수준으로 한정 (실제 저장은 안 되고 화면에서만 동작) 바로 운영해도 되는 진짜 서비스 수준까지 확장 (정보를 실제로 저장하는 서버+데이터베이스까지 포함)
인증/로그인 로그인 기능 자체를 만들지 않음 이메일·소셜 로그인 기능을 필수 기능으로 알아서 추가
신청 처리 흐름 신청 버튼을 누르면 바로 "신청완료"로 끝 신청 → 스터디장이 승인/거절 → 정원 차면 자동으로 모집 마감 처리까지 설계
알림 기능 알림 기능은 아예 언급하지 않음 앱 안에서 알림을 받는 기능을 새로 추가
데이터 저장 방식 화면에서만 잠깐 기억하는 임시 데이터 (새로고침하면 사라짐) 실제로 계속 저장되는 데이터베이스 설계 (회원/스터디/신청/알림 정보를 나눠서 저장)
화면/디자인 디테일 화면 분위기, 카드 배치, 색상, 빈 화면 문구까지 아주 구체적으로 지정 모바일에서도 잘 보이게 하라는 정도만 언급하고 세부 디자인은 생략
코드 설명(주석) 방식 기획자도 이해할 수 있도록 코드에 한글 설명을 꼭 넣게 함 코드 설명은 최소화하고 이름도 영어로 짓게 함 (정반대 방식)
화면 이동 방식 화면 전환 방식을 개발자가 편한 대로 유연하게 선택 가능 웹사이트 주소(URL) 체계를 실제 서비스처럼 고정된 방식으로 구성
개발 진행 방식 결과물 중심 — 다 만든 뒤 사용법·화면 구성·기능 설명을 정리해서 보여줌 과정 중심 — 데이터 구조부터 먼저 제안하고 확인받은 다음 진행하는 단계별 방식
Codex는 "일단 확인해볼 시제품"으로, Claude는 "바로 운영할 실제 서비스"로 받아들였다. 같은 한 줄 요청도 AI마다 어디까지 만들지 판단하는 기준이 다르다.

※ 다만 이는 하나의 실험 사례이며, AI별 결과 방향은 요청 맥락과 프롬프트 조건에 따라 달라질 수 있다.
